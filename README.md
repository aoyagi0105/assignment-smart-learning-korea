# assignment-smart-learning-korea

## Part A. 서버 설계

### 작업 단계

#### 1. DB 스키마 설계
**목표** : 조직별 업체 설정 및 인증 정보를 안전하고 유연하게 저장할 구조 설계  

**산출물** : 데이터베이스 ERD, 테이블 정의서  

**핵심 사항** : 업체마다 필요한 인증 정보(AccessKey, APIKey 등)가 다르므로, 별도의 테이블을 만들고 그 안에 JSONB 타입 컬럼을 만들어 유연하게 데이터를 저장한다

---

#### 2. 메시징 어댑터 인터페이스 정의 및 구현
**목표** : 업체마다 다른 API 규격(인증 방식, 필드명 등)을 공통 인터페이스로 통합  

**산출물** : 공통 인터페이스 및 업체별 어댑터 클래스  

**핵심 사항** : CloudMsg의 HMAC 서명 생성 로직이나 SendTalk의 Bearer 인증 처리를 각각의 어댑터 내부로 격리

---

#### 3. 메시지 라우팅 및 발송 로직 구현
**목표** : 메시지 발송 요청이 들어오면 조직 설정을 조회하여, 어댑터를 호출하는 로직 구축  

**산출물** : 어댑터 주입 로직, 라우팅 로직  

**핵심 사항** : DB를 조회하여, 어댑터로 연결후 메시지 전송이 가능

---

#### 4. 연동 검증 API 구현
**목표** : 화면(Part B)에서 입력한 설정값을 저장하고, 실제로 연동이 잘 되었는지 확인하는 API를 만듬  

**산출물** : 설정 관리 API 명세서 및 연동 테스트 결과 반환 로직  

**핵심 사항** : 설정값 저장뿐 아니라, 실제 업체 API를 호출하여 현재 입력된 데이터로 발송이 가능한가를 응답해야함

---

### 기술적 결정 사항

#### 1. 고민 : 업체마다 인증정보가 다른데 컬럼을 어떻게 구성할 것인가

**대안**

A : 업체별로 필요한 인증 정보 칼럼을 모두 생성  
B : 공통 정보는 칼럼으로, 가변정보는 JSONB로 만든다  

**선택과 근거** :  
B 선택 / A의 경우, 타입과 제약 조건을 명확이 걸수있어 데이터 안정성이 높고, 필드를 쉽게 조회할 수 있지만 새로운 업체가 추가될때마다 entity 및 테이블 수정을 해야하며, null컬럼이 많아진다.  

B의 경우, 데이터 안정성이 떨어지고, 특정 필드를 조회하는 문법도 까다로워지지만, 업체 추가마다 테이블과 entity를 수정하지 않아도 되고 null 컬럼이 비약적으로 많아지지도 않는다.  

또한 데이터 안정성은 서버나 프론트엔드에서 검사할 수 있다.

---

#### 2. 고민 : 업체마다 상이한 인증 방식(HMAC vs Bearer)과 API 규격을 메시지 발송 로직과 어떻게 분리할 것인가?

**대안**

A : 발송 로직 내 if-else 분기 처리  
B : 인터페이스 기반 어댑터 패턴  

**선택과 근거** :  
B 선택 / A의 경우, 구현 속도가 빠르고 업체수가 늘어나지 않고 소규모 프로젝트라는 가정하에 유리하다. 하지만 이후 업체가 늘어나게 될겨우 로직이 비대해지고 복잡해질수있다.  

B의 경우, 어댑터 코드를 작성하므로 특정 업체를 대상으로 테스트하기 편하고, 업체가 늘어나거나 특정 로직 수정이 필요할때 수정이 간편해진다. 또한 인터페이스 설계 및 클래스 생성을 위한 파일이 별도로 만들어지므로 가독성이 늘어난다는 장점이 있다.

---

### 리스크 및 대응

#### 1. 위험 요소 : 업체별 전송 제한( Rate Limits ) 초과

**영향 범위** : CloudMsg(200회/초), SendTalk(100회/초) 등 업체별 상이한 제한 초과 시 발송 중단 및 429 에러  

**완화 전략** : 어댑터 별로 임계치를 설정하여 발송 속도를 자동 조절. 초과된 요청은 버리지 않고 대기 후 순차적으로 재시도 처리

---

#### 2. 위험 요소 : 외부 API 장애 및 타임아웃

**영향 범위** : 한쪽 메시지 전송 업체의 서버 장애 시, 서버 전체의 스레드 고갈로 이어지는 연쇄 장애 발생. 다른 한쪽 메시지 전송 업체로도 메시지를 보낼수 없게됨  

**완화 전략** : 일정 횟수 이상 메시지 전송이 실패할 경우, 해당 업체로의 연결을 차단하고 일시 점검 상태로 전환. 연쇄 장애를 막을 수 있음.

---

## Part B. 화면 설계

### 화면흐름

1. 문자 발송 업체를 선택  
2. 선택한 업체에 따른 인증 정보 입력  
3. 테스트를 위해 각 업체별 메시지 타입에 필요한 정보 입력 (필수값 뿐만아니라 비필수값도 전부 입력 가능하도록)  
4. 입력한 정보로 실제 테스트 메시지 발송 및 결과 확인  
5. 테스트 성공 시에만 변경 사항을 시스템에 반영  

---

### UI 결정 사항

#### 1. 고민포인트 : 발신 번호, 수신 번호의 번호형식

**대안 비교**

A : `"01012345678"` 형식으로 통일  

B : CloudMsg API의 경우 `"+821012345678"` / 센드톡 API의 경우 `"01012345678"` 형식으로 작성  

**선택과 근거** :  
A 선택 / 일반적인 사용자의 경우 `"01012345678"` 형식에 익숙해져있기 때문. 화면에서는 편하게 입력하게 하되, 프론트엔드에서 규격에 맞게 변환하여 서버에 전송

---

#### 2. 고민포인트 : 가격과 Rate Limits의 예시 비교표를 작성할것인가

**대안 비교**

A : 예시 비교표를 작성  

B : API의 URL을 첨부. 대신 대략적인 가격과 Rate Limits을 표시  

**선택과 근거** :  
B 선택 / A의 경우 가격 변동에 따른 업데이트가 수시로 이루어져야하며, 업데이트가 이루어지지 않을 경우 사용자가 가격을 헷갈릴 수 있기 때문.  

B의 경우 사용자가 직접 가격을 찾아야하는 불편함이 있지만, API의 최신 가격을 알 수 있고, 대략적인 가격을 표시함으로써 일정 가격을 예상가능하게 한다.

---

### 예외 / 엣지 케이스

#### 1. 예약 발송 중 업체 변경시 생기는 에러

**상황** : 대량의 메시지를 예약 발송 하던 중 업체 변경시, 바뀐 업체의 템플릿이나 요구 사항에 맞지 않아 에러가 발생할 수 있다.  

**ui 대응 방안** : 관리자가 변경 저장 버튼을 눌렀을 때, 예약 발송 테이블에서 현재 시간과 발송 시간이 겹치는 데이터가 존재하면, 예약 메시지가 발송 실패 할 수 있음을 모달로 알린다.

---

#### 2. 인증 정보 입력 시 공백이 들어간 에러

**상황** : key 등 인증 정보를 복사해 오면서 공백이 포함될 수 있다.  

**UI 대응 방안** : 인증 정보에 공백이 추가되었을 경우, alert 혹은 모달을 통해 공백이 추가되었음을 알린다.

## AI 활용(제미나이 사용)
### 질문 및 결과
#### 
**질문** : 메시지 대량 발송 시 특정 업체 서버가 느려지거나 터졌을 때, 우리 서버 전체가 마비되는 '연쇄 장애'를 어떻게 막을 수 있는가?
**결과** : 서킷 브레이커(Circuit Breaker) 및 벌크헤드(Bulkhead) 패턴을 도입하여 시스템 안정성을 강화하는 완화 전략을 수립할 수 있었습니다.

#### 
**질문** : 인증 정보를 저장할 때 업체별로 컬럼을 다 만들 것인가, 아니면 JSONB를 쓸 것인가? 각각의 장단점은?
**결과** : AI와 각 방식의 인덱싱 효율, 유지보수 편의성을 비교 분석한 후, B2B SaaS 환경에 적합한 **하이브리드 스키마(JSONB 활용)**를 최종 결정함.
